# Web Selection Clip To Coto

This document explains the implementation of the in-app browser feature that clips selected web content into a Coto draft.

## Overview

The feature is available only in browser windows. When the user selects content in the browser content webview, Cotoami shows a small `Clip` button above the selection. Clicking the button captures the current selection, converts the selected HTML into Markdown, opens the browser timeline/editor pane, and replaces the current flow draft with a preview-mode Coto draft.

The generated draft has this shape:

```markdown
converted selected content

[Page Title](https://example.com/page)
```

If the editor already contains draft content, clipping replaces it unconditionally. The browser timeline uses window-local drafts, so clipped content is not persisted as a reusable saved draft.

## Webview Roles

The browser uses two webview roles:

- `browser-shell-*`: trusted local Cotoami UI. It owns the toolbar, timeline pane, editor, permission policy, and draft replacement.
- `browser-content-*`: the child webview that displays arbitrary remote web pages. It receives an injected selection script.

The visible `Clip` button is rendered inside the content webview, not in the shell DOM. This is intentional: the content webview is a native child view, so shell DOM elements cannot reliably appear above it with CSS `z-index`.

## Selection Script

The selection script is generated by `browser_selection_script` in `tauri/src/commands/browser.rs` and injected with `WebviewBuilder.initialization_script` when the content webview is created.

The script:

- listens for selection changes, mouseup, keyup, scroll, resize, and blur;
- reports lightweight selection state only during selection updates;
- does not send selected HTML while the user is dragging/selecting;
- renders the content-webview `Clip` button when the shell says the selection is valid;
- captures selected text, selected HTML, page title, page URL, and selection rect only when the button is clicked.

The script reports selection state through:

- `browser_selection_state`
- `browser_selection_capture`

The Rust side validates that the command came from the expected `browser-content-*` webview before forwarding events to the shell.

## IPC Notes

Remote pages in the content webview cannot safely use the same assumptions as local app pages. Some pages reject Tauri's custom protocol IPC fetch path, producing errors such as blocked `ipc://localhost/...` requests.

The injected script therefore nudges Tauri's command bridge to use Wry's native `window.ipc.postMessage` fallback for this selection bridge. It temporarily intercepts the custom-protocol IPC fetch and suppresses only the expected fallback warning, then restores the page's original `fetch` and `console.warn`.

This keeps the selection bridge quiet on strict websites while still using Tauri commands and capability checks.

## Clip Button Rendering

The shell decides whether the button is allowed, then calls:

```rust
browser_set_selection_clip_overlay
```

That command dispatches an event into the content webview. The injected script shows or hides the button at the selection rect.

The button label is localized with `BrowserShell_clipSelection`. The icon is the Cotoami `logomark.svg`, embedded as inline SVG markup in the injected button. It is not loaded with an `<img src="data:...">` because some sites reject `data:` images through their Content Security Policy.

## Capture Flow

The high-level flow is:

1. The content script detects a non-empty top-level selection.
2. It sends `browser_selection_state` with `has_selection`, URL, and viewport rect.
3. Rust validates the source webview and emits `browser-selection-state`.
4. `BrowserShell` checks:
   - clipping is allowed in the current app context;
   - the URL is current, not stale;
   - the selection has a rect.
5. `BrowserShell` calls `browser_set_selection_clip_overlay` to show the button inside the content webview.
6. The user clicks `Clip`.
7. The content script sends `browser_selection_capture` with selected text, selected HTML, title, URL, and rect.
8. Rust validates and emits `browser-selection-capture`.
9. `cotoami.browser.App` converts the captured content to Markdown and replaces the flow draft.

The shell also hides or invalidates the button on deselection, navigation, reload, stale URL, scroll/resize/blur, Escape, no database, or no posting permission.

## Markdown Conversion

Markdown conversion lives in `ui/src/main/scala/cotoami/browser/WebClipMarkdown.scala`.

The conversion pipeline uses:

- `rehype-parse` to parse selected HTML as a fragment;
- a local rehype AST pass to resolve relative URL attributes against the page URL;
- `rehype-remark` to convert HTML AST to Markdown AST;
- `remark-stringify` to serialize Markdown.

The URL normalization pass currently handles common URL-bearing attributes:

- `a[href]`, `area[href]`, `link[href]`
- `img[src]`, `source[src]`, `video[src]`, `audio[src]`, `track[src]`, `iframe[src]`, `embed[src]`, `script[src]`
- `object[data]`

This is important because clipped Markdown is rendered later without the original page base URL. For example:

```html
<a href="/docs/intro">Intro</a>
<img alt="Logo" src="../images/logo.png">
```

from `https://example.com/docs/page/index.html` becomes:

```markdown
[Intro](https://example.com/docs/intro)
![Logo](https://example.com/docs/images/logo.png)
```

If HTML conversion is blank, duplicated, or clearly worse than the selected plain text, the converter falls back to selected text. It then appends a safely escaped source link, using the page title when available and falling back to host/URL.

## Draft Replacement

Draft replacement is handled by `SectionFlowInput.Msg.ReplaceCotoDraft`.

When clipping, the browser app dispatches this message with `preview = true`. The handler:

- forces the flow input back to a plain `CotoForm.Model`;
- replaces `contentInput`;
- clears summary, media, date, and geolocation fields;
- opens/unfolds the editor;
- enables preview mode;
- persists only when draft persistence is enabled.

In the browser timeline, `persistDraft = false`, so clipped drafts are window-local.

## Tauri Permissions

Tauri v2 command access is deny-by-default. A webview can call a Rust command only if a capability grants a permission, and that permission maps to the command.

This feature uses two capability contexts.

### Remote Content Capability

`tauri/capabilities/browser-content.json` applies to:

```json
"webviews": ["browser-content-*"],
"remote": { "urls": ["http://*", "https://*"] }
```

It grants only:

```json
"allow-browser-selection-state",
"allow-browser-selection-capture"
```

Those permissions allow remote pages to call only:

- `browser_selection_state`
- `browser_selection_capture`

This keeps the remote page's power narrow. It can report selection state and send captured selection content, but it cannot navigate the browser, open panes, edit drafts, or post cotos directly.

### Local Shell Capability

`tauri/capabilities/migrated.json` applies to local windows:

```json
"windows": ["main", "browser-shell-*"],
"local": true
```

For this feature, the shell needs:

```json
"allow-browser-request-selection-capture",
"allow-browser-set-selection-clip-overlay"
```

Those permissions allow the trusted shell to call:

- `browser_request_selection_capture`
- `browser_set_selection_clip_overlay`

The shell gets these commands because it decides when clipping is allowed and controls the content-webview overlay. The remote content webview does not get them.

### Autogenerated Permission Files

The files under `tauri/permissions/autogenerated` are generated by Tauri from the command manifest in `tauri/build.rs`.

They are still source artifacts for this setup. Capability files reference permission identifiers such as:

```json
"allow-browser-selection-state"
```

The corresponding generated TOML file defines what that identifier means:

```toml
identifier = "allow-browser-selection-state"
commands.allow = ["browser_selection_state"]
```

Without the generated permission file, the identifier in the capability JSON is not connected to a Rust command and may fail capability validation or command access on a clean checkout.

The feature-specific generated permission files are:

- `browser_selection_state.toml`
- `browser_selection_capture.toml`
- `browser_request_selection_capture.toml`
- `browser_set_selection_clip_overlay.toml`

Other generated files correspond to pre-existing app commands that are listed in the same command manifest and granted by `migrated.json`.

sbt compile test
