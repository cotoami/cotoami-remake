# Tauri Permissions

This document explains how Tauri v2 permissions are used in Cotoami Desktop.

## Why Permissions Exist

Tauri v2 command access is deny-by-default. A frontend webview cannot call a Rust command just because the command exists in `invoke_handler`. The webview must also be covered by a capability that grants a permission for that command.

The relationship is:

```text
webview/window
  -> capability JSON
  -> permission identifier
  -> permission TOML
  -> allowed Rust command
```

For example:

```json
"allow-browser-set-selection-clip-overlay"
```

is a permission identifier. Its generated TOML file maps it to a command:

```toml
identifier = "allow-browser-set-selection-clip-overlay"
commands.allow = ["browser_set_selection_clip_overlay"]
```

Without the permission entry in a matching capability, Tauri rejects the frontend `invoke` call even if the Rust command is registered.

## Capability Files

Capabilities live in `capabilities/*.json`. They answer two questions:

- which windows/webviews does this capability apply to?
- which permissions are granted there?

Cotoami currently has two important capability contexts.

## Local App Capability

`capabilities/migrated.json` applies to local app windows:

```json
"local": true,
"windows": ["main", "browser-shell-*"]
```

This capability grants normal local app UI permissions. It includes Tauri core/plugin permissions such as:

```json
"core:default",
"core:window:allow-set-title",
"core:webview:allow-create-webview-window",
"dialog:default",
"shell:default"
```

It also grants app-defined command permissions such as:

```json
"allow-open-browser-window",
"allow-browser-navigate",
"allow-open-database",
"allow-connect-to-servers"
```

These `allow-*` identifiers are not arbitrary strings. Each one is defined by a TOML file under `permissions/autogenerated`.

For the Web Selection Clip feature, the local browser shell needs:

```json
"allow-browser-request-selection-capture",
"allow-browser-set-selection-clip-overlay"
```

Those let the trusted browser shell ask the content webview to capture the current selection and show/hide the visible Clip overlay.

## Remote Browser Content Capability

`capabilities/browser-content.json` applies to remote pages loaded in child browser content webviews:

```json
"local": false,
"webviews": ["browser-content-*"],
"remote": { "urls": ["http://*", "https://*"] }
```

This capability is intentionally narrow. It grants only:

```json
"allow-browser-selection-state",
"allow-browser-selection-capture"
```

Those allow the injected selection script in a remote web page to call only:

- `browser_selection_state`
- `browser_selection_capture`

The remote page can report selection geometry and captured selected content, but it cannot navigate the browser, open windows, edit drafts, access databases, or post cotos.

This separation is important because `browser-content-*` displays arbitrary websites, while `browser-shell-*` is trusted local Cotoami UI.

## Autogenerated Permission Files

The files under `permissions/autogenerated` are produced by Tauri from the app command manifest in `build.rs`.

`build.rs` lists app-defined commands:

```rust
tauri_build::AppManifest::new().commands(&[
    "open_browser_window",
    "browser_attach",
    "browser_selection_state",
    "browser_selection_capture",
    // ...
])
```

Tauri uses this list to generate permission definitions like:

```toml
[[permission]]
identifier = "allow-browser-selection-state"
commands.allow = ["browser_selection_state"]

[[permission]]
identifier = "deny-browser-selection-state"
commands.deny = ["browser_selection_state"]
```

The files are autogenerated, but they are still part of the permission model used by this app. If a capability references `allow-browser-selection-state`, the generated TOML file is what tells Tauri which Rust command that identifier means.

Do not edit these files by hand. Regenerate them by changing `build.rs` and running the Tauri build/check flow.

## Are All `allow-*` Permissions Needed?

Every `allow-*` permission in a capability should correspond to a command that the matching webview/window may call.

Not all of them are related to one feature. For example:

- `allow-browser-request-selection-capture` is for Web Selection Clip.
- `allow-open-database` is for opening databases.
- `allow-browser-navigate` is for browser navigation.
- `allow-show-window` is for showing app windows.

If a command is no longer invoked by the matching window/webview, its permission can be removed from the capability. But removing a permission while the UI still invokes that command will cause Tauri to reject the call at runtime.

## Security Guideline

Grant permissions to the narrowest webview/window set that needs them.

For local trusted UI, `migrated.json` can grant broader app commands. For remote web content, use a separate capability like `browser-content.json` and keep it very small.

The Web Selection Clip implementation follows this rule:

- remote content can report selection state/capture only;
- local shell controls overlay visibility, draft replacement, and editor behavior.
